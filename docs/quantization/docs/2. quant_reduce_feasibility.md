# 2. Quantizing the All-Reduce on WSE: Feasibility Analysis

> **Headline:** quantizing the *channel* of the K-tree all-reduce (transmit at INT8/INT4, dequantize before every `@faddh`, accumulate at FP16) does **not** compound error through tree depth — every add still happens at FP16. The recent TP-communication-compression literature (Apple 2024; EQuARX 2025; Flash Communication 2024) validates this empirically (MSE 0.0014, ≤0.5% perplexity drop on Llama-2 13B / Llama-3 70B) and theoretically (aggregated quantization errors across N branches → ≈Gaussian centered at 0 by Irwin-Hall). Applying a conservative bytes-proportional transit model to doc 1's measurements: **channel-only INT8 projects to 24-28% e2e reduction at P=32→100; outlier-aware INT4+BF16 (~4.2 bits avg) projects to 36-41%. Bcast-only INT8 is ~12-17% with zero intra-op accuracy risk.** What the user worried about — partial-sum quantization compounding error — is a real failure mode of *naive* low-precision accumulation, but it is not what the standard design does, and the design that *does* apply has been hardware-validated on TPU (EQuARX, 1.8× AllReduce speedup at MSE 0.0014).

---

## 2.1 Question

Doc 1 established that at P=100, the K-tree all-reduce consumes **67% of e2e** decode latency: ~3583 cycles in reduce phases 1+2 (intra-op, finishes the current GEMV at the K-tree root) and ~3505 cycles in phase-3 broadcast (inter-op, ships the final value to every PE for the next op). Both phases scale with payload bytes through the K-tree backbone. Halving payload bytes — by switching FP16 partial sums to INT8 — should attack both buckets at once.

The user raised a specific concern about doing this:

> "The reduce tensor is the partial result or the final result of the op. If we quantize the intermediate result, I suppose it would cause accuracy down."

This concern is well-founded for a *naive* implementation that quantizes once at the K-tree leaves and adds in low precision at every internal K-tree node — quantization noise enters the accumulator at every hop and compounds as `O(√depth)` in the unbiased-rounding case, worse with biased rounding. But the design pattern actually used in the TP-communication-compression literature does not work this way. This document explains the distinction, surveys the methods, and projects the speedup achievable on WSE under doc 1's measurement framework.

---

## 2.2 The "channel vs. accumulator" distinction (the answer to the accuracy concern)

**Naive design (what the user is worried about):**

```
leaf:        fp16_partial_sum → quantize → INT8
internal:    INT8 + INT8 → INT8     (low-precision adder, error compounds)
root:        INT8 → dequantize → fp16
```

**Standard literature design (channel-only quantization):**

```
leaf:        fp16_partial_sum → quantize → INT8 → fabric
internal:    INT8 ← fabric → dequantize → FP16
             FP16 + FP16 → FP16              (full-precision adder)
             FP16 → quantize → INT8 → fabric (next hop)
root:        INT8 ← fabric → dequantize → FP16
```

The two are not the same. In the standard design, **every `@faddh` runs at FP16, exactly like the current `comm_pe.csl` does today**. Only the *transmitted bytes* are reduced. The K-tree's accumulator never sees a low-precision number. Quantization noise enters once per hop on the way *out* of each internal node (re-quantization before send), but it does not feed forward through the addition operation — it is bounded by the per-block scale at every hop independently.

This is the design pattern in:

- **EQuARX (Google, arXiv:2506.17615)** — explicitly: *"we quantize the shard, send the quantized shard and the corresponding metadata (i.e. scale factors), dequantize the received shard (Dq) to higher precision (e.g. FP32), and perform the addition."* INT8 channel, FP32 accumulator. Hardware-measured **1.8× AllReduce speedup at MSE 0.0014**, vs. naive FP8 (which compounds error) at MSE 0.13 — **two orders of magnitude better quality**.
- **Flash Communication (arXiv:2412.04964)** — INT4/INT8 channel with FP16 accumulation between phases. Llama-3 70B perplexity unchanged at INT8.
- **Apple's "Towards Low-bit Communication" (arXiv:2411.07942)** — INT4 channel + 1/64 features kept at BF16 (outliers); dequantize before sum on every receiving device.

The user's concern is therefore valid as a critique of the *naive* design but does not apply to the design the literature actually uses, which is the design we should adopt for WSE.

## 2.3 Why per-hop re-quantization noise stays bounded

Even in the channel-only design, re-quantization happens at every internal K-tree node before the next send. Over `depth ≈ log₂(group_num) + log₂(pe_num_p_group)` hops, the question becomes: does the per-hop rounding noise compound?

**Three independent bounds on the error growth:**

### 2.3.1 Stochastic rounding makes per-hop noise unbiased

Stochastic rounding (Gupta et al. 2015; QSGD, arXiv:1610.02132) — rounding a value `x ∈ [k, k+1]` *up* with probability `x − k` and *down* with probability `1 − (x − k)` — produces an unbiased estimate: `E[round(x)] = x`. With per-hop noise zero-mean, the variance after `D` hops grows as `D × σ²` (linear in depth, not exponential), and the RMS error grows as `O(√D)`. For the WSE K-tree at P=100 with `group_num = 10`, depth ≈ 4+4 = 8 hops; even 7 internal re-quantizations give error ≈ 2.6× single-hop error — for INT8 (256 levels with per-block scale) this is well under the FP16 baseline noise floor (FP16 has 11 mantissa bits = ~3-4 ULP noise per operation).

### 2.3.2 The all-reduce *averages out* leaf-level rounding noise (Irwin-Hall)

Apple's TP-comm paper (arXiv:2411.07942) makes the following observation: when N devices each contribute a partial sum quantized with independent error `ε_i ~ Uniform(−Δ/2, +Δ/2)`, the aggregate error after summation `Σ ε_i` follows the Irwin-Hall distribution (sum of N iid uniforms), which approaches `N(0, N·Δ²/12)` by CLT for large N. Crucially, the **relative** error of the *sum* is `√(N·Δ²/12) / Σx_i ∝ Δ/√N` — so adding more devices *reduces* the relative quantization error of the all-reduce result, rather than increasing it. This is a per-leaf quantization argument, not a per-hop re-quantization argument, but it stacks with §2.3.1 and gives the all-reduce structure an inherent denoising property.

**Caveat — not formally proved.** The Apple paper invokes Irwin-Hall as intuition, not theorem; they do not prove the bound rigorously and the argument assumes independent errors per branch (which only fully holds if leaf quantizations are done with independent random offsets, e.g. per-PE stochastic rounding). With deterministic round-to-nearest, the errors are correlated and the bound weakens.

### 2.3.3 Empirical evidence: ring AllReduce stddev does not grow with N

main-horse's "Why reduction precision matters" blog post reports an unexplained empirical finding: across world sizes 2–1024 GPUs, the BF16 ring AllReduce output stddev does *not* increase with N. This is consistent with §2.3.2 (the all-reduce averaging out per-leaf noise) but the author flags they "don't understand why reduction precision matters" — i.e., the empirical evidence is more favorable than theory predicts. Useful as a sanity check that the channel-only INT8 design will not catastrophically fail in practice.

### Net implication for WSE

For doc 1's sweep configurations (depth ~6-8 hops, INT8 with per-block scale, optional stochastic rounding), the channel-only design's per-K-tree-call output error should be well below FP16's existing rounding noise floor. Empirical validation against an FP16 reference (separate task; see §2.8) is required to confirm in our specific kernel, but the literature gives no reason to expect catastrophic accuracy degradation.

## 2.5 Speedup projection

### Cost model

K-tree latency at each phase has two components:
- **Transit-bound** (bytes-proportional) — depends on payload byte count, scales with K-tree depth × per-hop transit time. Quantization helps here.
- **Per-PE-local** — reconfig (`reset_routes`-equivalent), the `@faddh` itself, fabric setup. Quantization does not help here.

From doc 1, P=32 baseline:
- `comm_mean` = 5621 cycles = `reduce_mean` (3512) + `bcast_mean` (2109)
- Reconfig ≈ 1188 cycles per step (3 × `op:reconfig_x` ~200 + 3 × `op:reconfig_y` ~196) — these are *inside* `reduce_mean` per doc 1's definition.
- **Transit-bound** = (reduce − reconfig) + bcast = (3512 − 1188) + 2109 = **4433 cycles** at P=32.
- Per-PE-local non-quantizable cost = reconfig (1188) + ~half of the FP16 add cost (small). Treat as ~1188 for the conservative model.

**Conservative bytes-proportional model:** transit-bound cycles scale linearly with payload bytes. INT8 (8 bits) → 0.5× transit; INT4+BF16 outlier-aware (~4.2 bits avg) → 0.26× transit. The non-transit component is held constant.

This is conservative because it ignores possible *additional* wins from reduced fabric router contention and queue depth at lower payloads. It is *aggressive* only in assuming dequant compute overhead is fully hidden by streaming DSDs — EQuARX validates this assumption holds within ~10% on TPU (achieves ~90% of theoretical speedup); the assumption may be slightly tighter on WSE because per-PE workload is much smaller.

Applying the model to all 6 configs from doc 1's `overhead_vs_P.csv`:

|---|---|---|---|---|---|
| 32 | 9083 | 4433 | **6866 (−24%)** | 8029 (−12%) | **5803 (−36%)** |
| 40 | 9248 | 4557 | **6969 (−25%)** | 8141 (−12%) | **5876 (−36%)** |

(`A` halves transit; `B` halves only the bcast portion; `C` multiplies transit by 0.26 ≈ 4.2/16, the bit-budget ratio of Apple's INT4+1/64-BF16 scheme vs. FP16. Reconfig + compute held constant in all rows.)

### Reading the table

- **Channel-only INT8 (A) gains compound with P.** At P=32 the win is 24% e2e; at P=100 it grows to 28%. This is because transit is a larger share of e2e at higher P (bcast scales with grid diameter while compute and reconfig stay flat).
- **Bcast-only INT8 (B) gains also compound with P** — 12% → 17% — for the same reason: bcast is the part that grows.
- **Outlier-aware INT4+BF16 (C)** dominates at every P (36% → 41%), but requires solving the CSL fabric metadata problem (§2.7).
- **The crossover where bcast-only beats nothing-changed is everywhere** — even at the smallest P, the simplest possible quantization (B) saves 12% e2e at zero accuracy risk.

### Sensitivity to model assumptions

Two assumptions could weaken the projection:
1. **Bytes-proportional transit may overstate at small payloads.** WSE fabric routers have a fixed per-hop latency (~3 cycles minimum) plus a payload-proportional component. At very small payloads (the `dim_p_pe = 8` element vector × FP16 = 16 bytes per primitive), the fixed component is non-negligible. If the fixed component is, e.g., 30% of per-hop transit, then INT8 wins only 0.7 × 0.5 + 0.3 = 0.65× transit instead of 0.5×, and the e2e reduction at P=100 drops from 28% to ~20%.
2. **Dequant compute overhead may not fully hide.** EQuARX achieves 90% of theoretical speedup on TPU via deep pipelining of microshards; on WSE, smaller per-PE workload may leave less room to hide dequant. If dequant adds, say, 20% to per-hop cost, INT8 wins only 0.6× transit, and the P=100 reduction drops to ~22%.

The honest range for **A at P=100 is therefore 20-28% e2e reduction in the realistic-assumptions case**, not a single number. The table's 28% is the upper-end (most generous-to-quantization) reading consistent with EQuARX's TPU validation; ~20% is the realistic-conservative lower bound if either fixed-router-cost or dequant-overhead is moderately worse than expected. A pathological floor of ~12-15% applies only if *both* assumptions fail badly (e.g. 30% fixed router cost + 20% dequant overhead compounded multiplicatively); EQuARX's measurements suggest this is unlikely.

**Caveat: model-size gap.** Apple's bounds are on Llama-2 13B / Gemma-2 27B / Mistral NeMo 12B; WaferLLM targets a TinyLlama-class ~1.1B model. The accuracy bounds therefore do not directly apply. The gap is *favorable* (smaller models have simpler activation distributions, fewer outliers, are typically easier to quantize), but unvalidated. A real validation would require running the quantized kernel against weights that were either (a) loaded from a TinyLlama checkpoint, or (b) calibrated on a small dataset — both out of scope here. **Bcast-only (B) is the most robust to this gap** because it does not change anything intra-op; the K-tree root computes the exact FP16 reduction, and only the *between-op handoff* is quantized — equivalent to standard INT8 activation quantization which is the most well-studied part of the LLM quantization literature.

## 2.7 WSE-specific feasibility notes

A peek at `test/decode_interop/src/comm_lib/comm_pe.csl` lines 491–1032 confirms the streaming pattern: each `all_reduce_*` primitive uses fabric DSDs (`reduce_1st_*_recv`, `reduce_2nd_*_send`, `bdcast_*`) directly as operands of `@faddh` / `@mov16`. The fabric DSD streams data from the fabric color into the operation; there is no intermediate buffer at the recv side.

### Dequant insertion

For methods A and C, dequant must be inserted between fabric recv and `@faddh`. This breaks the streaming pattern and requires:

1. A local FP16 scratch buffer (size = vector length per primitive; small, e.g., 8 elements at P=32 → 16 bytes).
2. A scale factor received over a separate fabric color (or interleaved with data; both have CSL support).
3. The CSL ops `@i2f`, `@fmul` to apply the scale, then the existing `@faddh`.
4. After the add, `@f2i_s` (round-to-nearest) or `@f2i_s` + an additive noise channel for stochastic rounding, then send INT8 over the next K-tree edge.

This adds approximately 3-5 instructions per fabric recv site × ~12 recv sites per primitive × 6 primitives ≈ 200 instructions of new CSL. Manageable but non-trivial. EQuARX's pipelined design suggests the per-hop dequant can be partially overlapped with neighbor receives via streaming, so the wall-clock overhead may be smaller than the instruction count suggests.

### For method B (bcast-only)

The change is much smaller: only 6 sites total. Each `all_reduce_*` primitive's bcast block looks like:

```csl
// Broadcast (current FP16)
prt.fmt_with_coords("bcast_start:all_reduce_bsz\n", .{});
if (pe_id == root_2nd_phase) {
    @mov16(bdcast_send_bsz, vector_buf_dsr);
} else {
    @mov16(vector_buf_dsd_bsz, bdcast_recv_bsz);
}
```

Quantization version (sketch):

```csl
// Broadcast (INT8)
prt.fmt_with_coords("bcast_start:all_reduce_bsz\n", .{});
if (pe_id == root_2nd_phase) {
    quantize_block_fp16_to_int8(vector_buf_dsr, bdcast_send_bsz_int8, scale_send);
} else {
    dequantize_block_int8_to_fp16(bdcast_recv_bsz_int8, vector_buf_dsd_bsz, scale_recv);
}
```

with a scale broadcast from root over a small fabric color before the data. Roughly 4-6 lines per site × 6 sites = ~30 lines of new CSL. **B is the obvious first experiment** — it touches the smallest amount of code, has zero intra-op accuracy risk, and validates the measurement plumbing (parser already extracts `bcast:*` markers separately, so the same `compute_breakdown.py` works unchanged).

### Open question: scale transport on the fabric

For both A and B, the per-block scale needs to travel alongside the data. Two options:

1. **Separate fabric color for scale.** Add one new color, sized 1 FP16 per block. Cheap but adds routing config.
2. **Interleaved metadata in the data stream.** Standard MX format puts the scale in a fixed metadata slot. Requires CSL DSD with mixed dtype reads — unclear how natural this is in CSL 1.4.

This needs to be resolved before A or B can be implemented, but does not affect the accuracy / projection arguments.

### Stacking with Decode_pipeline

Doc 1 §1.4 noted that Tetris's `Decode_pipeline` (static-color, dedicated comm PEs, no runtime reconfig) saves an additional ~12% e2e by eliminating the 1188-cycle reconfig cost. **This stacks orthogonally with quantization.** Combined projection at P=100: 28% (A) + 12% (Decode_pipeline) ≈ 40% e2e reduction (multiplicative model: `0.72 × 0.88 = 0.63`, i.e., **37% reduction**). Outlier-aware C + Decode_pipeline: `0.59 × 0.88 = 0.52`, i.e., **48% reduction**.

## 2.8 Recommendation

Implementation ordering for a future task (this task does not implement):

1. **Stage 1: Method B (bcast-only INT8).** Smallest code change, zero intra-op accuracy risk, validates the measurement plumbing. Expected win: 12-17% e2e at P=32→100. Use this stage to (a) confirm the bytes-proportional transit model holds on WSE and (b) work out the scale-transport mechanism.
2. **Stage 2: Method A (channel-only INT8 across all 3 K-tree phases).** Reuse the scale-transport mechanism from stage 1. Expected win: 24-28% e2e at P=32→100 (replaces stage 1's contribution; not additive). Ablate against stage 1 to confirm where the savings come from.
3. **Stage 3 (optional): Method C (outlier-aware INT4 + BF16).** Only if A's accuracy/perf isn't enough. Adds the outlier-mask transport problem on top. Expected win: 36-41% e2e.
4. **Stage 4 (optional): Method F (stochastic rounding + error feedback).** Layer on top of A or C. Expected accuracy improvement; speedup unchanged.

**Out of scope for this analysis:** any choice between stages, any commitment to implement. The next task (3.X) will choose a stage and write its implementation plan.

## 2.9 Limitations and open questions

1. **Bytes-proportional transit model is an approximation.** §2.5 sensitivity analysis: actual win for INT8 (A) at P=100 is in [20%, 28%] depending on fixed-vs-variable router latency and dequant overhead.
2. **Accuracy bounds in §2.6 are extrapolated from larger models.** Apple/Flash/EQuARX evaluate on 7B–70B models; WaferLLM targets ~1.1B. The gap is favorable in expectation but unvalidated.
3. **Apple's Irwin-Hall argument is intuitive, not formally proved.** The independence-of-error assumption requires either stochastic rounding or per-PE random offsets to hold strictly; with deterministic round-to-nearest the bound weakens.
4. **Stochastic rounding requires a noise source.** WSE's CSL fabric does not natively expose a PRNG; implementations would either pre-compute random offsets at host and ship them, or use a low-cost per-PE LFSR. Cost not estimated.
5. **Outlier transport on WSE fabric (method C) is unsolved.** Standard MX format reserves a metadata slot; CSL DSD support for mixed-dtype reads is unclear in SDK 1.4.
6. **No empirical validation in this analysis.** All projections are model-based + literature-cited. A future task should run B at minimum and confirm the cycle reduction and (separately) the numerical equivalence vs. FP16.
7. **Group_num sweep at fixed P is a separate axis.** Doc 1 §1.6 noted that the K-tree shape (`group_num`) modulates bcast cost; the optimum shape may shift with payload size. Not addressed here.
